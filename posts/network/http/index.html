지금은 HTTP 시대!#

HTTP#

HyperText Transfer Protocol

HTTP는 문서간의 링크를 통해 연결할 수 있는 HTML을 전송하기 위한 프로토콜로 시작되었다.1
하지만 현재는 아래와 같이 거의 모든 데이터를 HTTP 프로토콜에 담아 전송한다.

  • HTML, TEXT
  • IMAGE, 음성, 영상, 파일
  • JSON, XML (API)
  • 서버간에 데이터를 주고 받을 때도 대부분 HTTP 사용

즉, 현재는 바야흐로 HTTP의 시대이다.

HTTP의 역사#

  • HTTP/0.9 1991년: GET 메서드만 지원, HTTP 헤더X
  • HTTP/1.0 1996년: 메서드, 헤더 추가
  • HTTP/1.1 1997년: 가장 많이 사용, 우리에게 가장 중요한 버전
  • RFC2068 (1997) -> RFC2616 (1999) -> RFC7230~7235 (2014)
  • HTTP/2 2015년: 성능 개선
  • HTTP/3 진행중: TCP 대신에 UDP 사용, 성능 개선

HTTP 특징#

HTTP의 특징 크게 4가지로 나눠보자면 아래와 같다.

  • 클라이언트 서버 구조
  • 무상태 프로토콜(Stateless), 비연결성
  • HTTP 메시지
  • 단순함, 확장 가능

하나씩 차례대로 알아보자.

1. 클라이언트 서버 구조#

HTTP의 첫번째 중요한 특징은 클라이언트/서버 구조(CS) 로 되어있다는 점이다.

위 예제를 해석하자면 아래와 같다.

  • Request Response 구조
  • 클라이언트는 서버에 요청을 보내고 응답 대기
  • 서버의 요청에 대한 결과를 만들어서 응답

굉장히 표면적이지만 단순하다.

하지만 클라이언트와 서버를 개념적으로 분리되면서 양쪽이 독립적으로 진화할 수 있다는 있다는 점은 굉장히 중요한 부분이다.

이렇게 비즈니스, 데이터 로직은 서버가 UI(화면)/UX(사용성)은 클라이언트가 담당하게 되었다.

우리는 트래픽이 폭주해도 서버 아키텍쳐와 백엔드 기술을 얼마나 더 고도화하고 진화시킬수 있는지에 대해서만 고민하면 된다.

2. 무상태 프로토콜(Statelesss)#

HTTP는 서버가 클라이언트의 상태를 보존하지 않는다.
우선 우리는 Stateful과 Stateless의 차이를 알아봐야 한다.

  • 상태 유지 - Stateful

    #

    고객: 이 노트북 얼마인가요?
    점원: 100만원 입니다.

    #

    고객: 2개 구매하겠습니다.
    점원: 200만원 입니다. 신용카드, 현금중에 어떤 걸로 구매하시겠어요?

    #

    고객: 신용카드로 구매하겠습니다.
    점원: 200만원 결제 완료되었습니다.

너무나 일상적인 대화다 그런데 예제를 조금 바꿔서 점원이 바뀐다면 어떨까?

  • 상태 유지 - Stateful, 점원이 중간에 바뀌면?

    #

    고객: 이 노트북 얼마인가요?
    점원A: 100만원 입니다.

    #

    고객: 2개 구매하겠습니다.
    점원B: ?

    #

    고객: 신용카드로 구매하겠습니다.
    점원C: ?

무언가 이상하다. 고객의 요청은 점원이 바뀜에 따라 문맥이 유지되지 않는다.
이를 다시 정리해보자면 아래의 예제와 같다.

  • 상태 유지 - Stateful, 정리

    #

    고객: 이 노트북 얼마인가요?
    점원: 100만원 입니다. (노트북 상태 유지)

    #

    고객: 2개 구매하겠습니다.
    점원: 200만원 입니다. 신용카드, 현금중에 어떤 걸로 구매하시겠어요 (노트북, 2개 상태 유지)

    #

    고객: 신용카드로 구매하겠습니다.
    점원: 200만원 결제 완료되었습니다. (노트북, 2개, 신용카드 상태 유지)

상태 유지 상태에서 점원이 바뀌게 되다는 것은 통신의 관점에선 장애가 발생한것과 다름이 없다.

상태 유지를 알아보았으니 이제 무상태성을 알아보자.

  1. 하이퍼텍스트(Hypertext)는 참조(하이퍼링크)를 통해 독자가 한 문서에서 다른 문서로 즉시 접근할 수 있는 텍스트이다. ↩︎
